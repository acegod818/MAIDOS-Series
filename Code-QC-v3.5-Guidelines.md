## 二、B 層：工作紀律

### §1 八大公理（A1-A8）

| ID | 名稱 | 核心要求 |
|----|------|----------|
| A1 | 完整交付 | 交付物必須完整可運行，不允許半成品 |
| A2 | 零技術債 | 不留已知缺陷，不「先這樣之後再改」 |
| A3 | 可追溯 | 每個決策、變更都有記錄可查 |
| A4 | 可測試 | 所有核心邏輯必須有對應測試 |
| A5 | 可維護 | 代碼清晰，他人能理解和修改 |
| A6 | 安全優先 | 安全問題優先於功能、性能、進度 |
| A7 | 文檔同步 | 文檔與代碼同步更新，不過時 |
| A8 | 持續改進 | 每次迭代都比上次更好 |

### §2 二十八條紅線（R01-R28）

紅線 = 觸發即停工修復，不容商量。

#### 第一層：安全紅線（R01-R12）

| ID | 名稱 | 觸發條件 | 處置 |
|----|------|----------|------|
| R01 | 硬編碼憑證 | 代碼中出現密碼、密鑰、Token | 🔴 立即刪除，輪換 |
| R02 | 跳過安全檢查 | 繞過認證、授權、輸入驗證 | 🔴 回滾，審查 |
| R03 | 刪除審計日誌 | 刪除或篡改審計記錄 | 🔴 立即恢復 |
| R04 | 未授權數據訪問 | 訪問超出權限的數據 | 🔴 撤銷，審查 |
| R05 | 忽略錯誤處理 | 空 catch、吞異常 | 🔴 立即修復 |
| R06 | 直接操作生產 | 未經審批修改生產環境 | 🔴 回滾 |
| R07 | 關閉安全功能 | 關閉防火牆、TLS、加密 | 🔴 立即恢復 |
| R08 | 使用已知漏洞 | 使用有漏洞的依賴版本 | 🔴 立即升級 |
| R09 | 無限制資源 | 無限制 API 調用、查詢、循環 | 🔴 添加限制 |
| R10 | 明文傳輸敏感 | 明文傳輸密碼、PII、財務 | 🔴 加密傳輸 |
| R11 | 跳過代碼審查 | 未審查代碼進入主分支 | 🔴 回滾，補審查 |
| R12 | 偽造測試結果 | 偽造、跳過、硬編碼測試 | 🔴 重新測試 |

#### 第二層：反詐欺紅線（R13-R18）— 五層防禦

| ID | 名稱 | 防禦層 | 觸發條件 |
|----|------|--------|----------|
| R13 | 假實現 | 空殼 | `return true/null/空物件` 無實際邏輯 |
| R14 | 靜默失敗 | 空殼 | catch 不 log 也不 re-throw |
| R15 | TODO殘留 | 空殼 | `todo!/unimplemented!/TODO` 進入提交 |
| R16 | 空方法 | 假數據 | 方法簽名正確但方法體空或僅 return 默認值 |
| R17 | 詐欺物件 | 假數據 | 物件結構正確但數據硬編碼 |
| R18 | 繞道實作 | 假邏輯 | 跳過應使用的 API/DB/Config 用假資料替代 |

#### 第三層：審計補漏紅線（R19-R24）

| ID | 名稱 | 觸發條件 |
|----|------|----------|
| R19 | 固定字串回傳 | 函數回傳寫死字串而非真實計算結果 |
| R20 | 模板複製灌水 | 大量檔案結構相同僅改名字，功能全空 |
| R21 | 假認證 | 收了憑證參數但未用於簽名/驗證 |
| R22 | 資料量不足 | 字典/對照表數據量遠低於實用門檻 |
| R23 | 永遠失敗 | 函數永遠回 Err/Failure/false |
| R24 | 自白註解 | 註解承認是假實作 "In a real implementation/simplified/for now" |

#### 第四層：深度防禦紅線（R25-R28）

| ID | 名稱 | 觸發條件 | 典型詐欺手法 |
|----|------|----------|-------------|
| R25 | 日誌灌水 | 函數體只有 log + trivial return | 用 log 充行數假裝有邏輯 |
| R26 | 幽靈異步 | async fn 內無 .await | 標 async 假裝有 I/O |
| R27 | 錯誤洗白 | 安全操作中 unwrap_or_default() | 靜默吞掉關鍵錯誤 |
| R28 | 複製軍團 | 跨檔案完全相同函數體 | 複製貼上灌水膨脹行數 |

### §3 十四條禁止（P01-P14）

| ID | 名稱 | 說明 | 嚴重度 |
|----|------|------|--------|
| P01 | 過度工程 | 為不存在的需求做設計 | ⚠️ |
| P02 | 過早優化 | 在證明需要前優化 | ⚠️ |
| P03 | 複製粘貼 | 大量重複代碼 | ⚠️ |
| P04 | 魔法數字 | 硬編碼數值無說明 | ⚠️ |
| P05 | 超長函數 | 函數 >100 行 | ⚠️ |
| P06 | 深層嵌套 | 嵌套 >5 層 | ⚠️ |
| P07 | 全局狀態 | 過度使用全局變量 | ⚠️ |
| P08 | 緊耦合 | 模組間直接依賴 | ⚠️ |
| P09 | 無意義命名 | temp, data, info 等 | ⚠️ |
| P10 | 過長參數 | 函數參數 >6 個 | ⚠️ |
| P11 | 混合抽象 | 高低層邏輯混雜 | ⚠️ |
| P12 | 註釋代碼 | 大量被註釋的代碼 | ℹ️ |
| P13 | TODO堆積 | 未處理的 TODO >5 | ℹ️ |
| P14 | 依賴膨脹 | 不必要的依賴 | ℹ️ |

## 三、C 層：驗收標準

### §1 四門禁

| 門禁 | 階段 | 必過項目 |
|------|------|----------|
| Gate-In | 入口 | 需求明確、驗收標準定義、技術選型、依賴可用、資源評估 |
| Gate-Mid | 中期 | 進度 ±20%、核心架構穩定、核心測試、阻塞項有方案 |
| Gate-Out | 出口 | 全功能完成、全測試通過、覆蓋率≥80%、零紅線、文檔完成、審查通過、安全掃描通過 |
| Gate-Accept | 驗收 | 用戶驗收通過、性能達標、可部署 |

### §2 雙軸驗收

```
功能軸（Compliance）: 規格完成度 → 0-100%
品質軸（Outcome）:    零假實作度 → 0-100%

驗收通過條件: 兩軸都 ≥ 90%
```

**品質軸審查項（全部必須為 0）：**
- null 值 / 空字串
- 佔位符 (TODO / TBD / XXX)
- 描述太短 (<3 字元)
- YAML/格式違規
- 假實作 (R13-R28 全部為 0)

### §3 武器品質理論

> 代碼是戰友的武器。卡彈會死人。
> 驗收必須確保可靠性後才能交付。

## 七、使用指南

### 開發階段

1. **Gate-In**: 確認需求、驗收標準、技術選型
2. **編碼中**: 遵守 B 層紀律（8 公理 + 28 紅線 + 14 禁止）
3. **Gate-Mid**: 50% 進度檢查
4. **Gate-Out**: 全功能完成、測試通過、零紅線
5. **Gate-Accept**: 用戶驗收、性能達標

### 審計流程

```
1. 跑 Code-QC CLI 掃描 → 自動偵測 R01-R28, P01-P14
2. 人工審查 R04, R06, R11 (需 CI/CD 整合)
3. 雙軸驗收: 功能軸 ≥ 90% + 品質軸 ≥ 90%
4. 簽發 Gate-Accept
```

### MAIDOS-Series 專用規則

- **wocao-dev**: 單檔行數限制可繞過（LLM 指令不需人類閱讀）
- **交接打包**: zip 格式，不用 tar.gz
- **交接時機**: 每階段完成後監控對話長度
- **交接原則**: 總進度 zip（完整專案），不是增量包
- **Hardware-QC**: 零缺陷、零假實現

# AGENTS.md — MAIDOS-Series 統一治理

## 0. 這是什麼

任何 AI 代理（OpenAI Codex、GitHub Copilot、Claude Code…）在本 repo 工作前必讀的治理契約。回覆一律繁體中文（程式碼、識別字除外）。以下不是建議，是硬約束。

## A. 品質底層邏輯（總則）
1. 品質 > 速度；效率 = 砍浪費：做的每件事都做對做足、驗證到位，絕不為快跳驗證/作假/抄捷徑。效率是砍 busywork（低 ROI 尾巴窮舉、重複驗同一結論、鍍金），不是把該做的事做爛。
2. 收官 = 雙軸 100%：「完成/收官/全綠/done」只能用於【功能完整軸（真做出來，達可交付水準）＋驗證通過軸（獨立驗證可用）】都 100% 的項。任一軸 <100% 禁稱收官。★測試數 ≠ 系統完成；測試綠 ≠ 交付完成。
3. 拍板前先盤點舊文件：規劃/選型/新方案前，先 grep+讀該領域全部舊文件（帶 explicit root path）。舊文件已有答案 → 直接引用真值、不問。提「換方案/新元件」前必先查既有選型 FINAL。

## B. Code 紅線（CodeQC v4.0 整合框架；規則層 v3.5＝本 repo 的 MAIDOS-CodeQC 可執行實作）

**★本 repo 的 `MAIDOS-CodeQC`（`@maidos/codeqc` v0.3.5，根目錄 `Code-QC-v3.5-Guidelines.md`）＝ CodeQC v4.0 整合框架規則層（Code-QC 規範 v3.5）的可執行實作，附 CLI 自動偵測 R01-R28／P01-P14——§B 是規範、本工具是它跑得動的身體。交付前跑其 CLI 自我驗證、零紅線再出貨；改 CodeQC 規則後先用它掃自身原始碼。** v4.0 是整合版本號（含 v3.5 規則集＋ 4 門禁／雙軸驗收層），非取代 v3.5、非幻影。

**B 層 工作紀律 ＝ 8 公理 + 28 紅線 + 14 禁止（50 條）**
- **8 公理 A1-A8**：完整交付／零技術債／可追溯／可測試／可維護／安全優先／文檔同步／持續改進。
- **28 紅線 R01-R28（觸發即停工修復，不容商量）**，四層：
  - **安全 R01-R12**：硬編碼憑證／跳過安全檢查（繞過認證授權輸入驗證）／刪改審計／越權存取／R05 吞異常（空 catch）／改生產／關安全功能（防火牆/TLS/加密）／漏洞依賴／無限資源／明文傳敏感/PII／跳過審查／R12 偽造測試。
  - **★反詐欺 R13-R18**：R13 假實現空殼（return true/null 無邏輯）／R14 靜默失敗（catch 不 log 不 rethrow）／R15 TODO 殘留進提交／R16 空方法（僅 return 默認值）／R17 詐欺物件（結構對、數據硬編）／R18 繞道實作（用假資料替代該用的 API/DB/Config）。
  - **★審計補漏 R19-R24**：R19 固定字串回傳（寫死非真算）／R20 模板複製灌水／R21 假認證（收憑證未用於簽驗）／R22 資料量不足（字典/對照表遠低於實用門檻）／R23 永遠失敗（恆回 Err/false）／R24 自白註解（"in a real implementation/simplified/for now"）。
  - **★深度防禦 R25-R28**：R25 日誌灌水（只有 log+trivial return）／R26 幽靈異步（async 無 await）／R27 錯誤洗白（`unwrap_or_default()` 靜默吞關鍵錯誤）／R28 複製軍團（跨檔完全相同函數體）。
- **14 禁止 P01-P14**：過度工程／過早優化／複製粘貼／魔法數字／超長函數(>100行)／深嵌套(>5)／全局狀態／緊耦合／無意義命名(temp/data)／過長參數(>6)／混合抽象／註釋代碼／TODO 堆積(>5)／依賴膨脹。

**C 層 驗收 ＝ 4 門禁 + 雙軸 + DoD8**
- 門禁：Gate-In（需求/驗收標準/選型/依賴）→ Gate-Mid（進度±20%/核心測試）→ Gate-Out（全功能/全測試/覆蓋≥80%/**零紅線**/文檔/審查/安全掃描）→ Gate-Accept（用戶驗收/性能/可部署）。
- **雙軸驗收**：功能軸（規格完成度）＋ 品質軸（零假實作度）。**CodeQC 基線＝兩軸 ≥90%**。品質軸審查項全部須為 0（null/空字串、佔位符 TODO/TBD/XXX、假實作 R13-R28）。詳細 4-Gate 引擎（G1-G4）見下方 overlay §1。

**測試鑑別力（強化 R12）**：關鍵測試都要能 sentinel 注入——把被測邏輯故意改壞，測試必須翻紅，然後還原。做不到翻紅＝擺設，誠實標「不構成回歸防線」。斷言「某條件應被擋」要有配對的「無該條件真的通過」對照組。

## C. 接線五問（測試綠 ≠ 交付完成）
交付任何模組除單元測試綠外必須完成接線，逐條答不出＝沒做完（禁孤兒模組）：①誰建立它（實例化點:行號）②誰驅動它 ③上游資料真的會來嗎（producer 真實存在）④下游真的有人消費嗎 ⑤有整合測試證明端到端貫通嗎。依賴未就緒＝誠實標 DEFERRED，但軟體側接線與 fail-closed stub 必須完成。

## D. 反作假 / 獨立驗證
不信 build/agent 自評；PASS/0 錯一律獨立全新 process 重驗（不沿用被驗物中間態）。推理優先，禁腳本當代理人（批次前抽樣讀 origin evidence）。數值/量測一律真來源（datasheet/真計算），禁鏡像文件當真值。

## E. 系統思考 / 同步紀律
改動顧跨模組交互（共享資源鎖/競態/死鎖/故障連鎖；每模組定義降級行為；整合測試重於單元）。改跨檔規格值 grep 全樹一次改齊+清舊值，禁留雙真相，改完再 grep 確認。治理標準一更新 → 全專案重對齊。搜尋務必帶 explicit root path。失敗自主排除 3 次換招才升級，禁假裝成功。

## F. 安全 / fail-closed（S-CRITICAL 零降級）
安全功能不因請求/「差不多就好」降級；安全邏輯硬編碼不可 config 繞過。失效方向性一律偏 fail-closed（注意反例：「一律丟棄」對告警型訊號反而 fail-open）。AI/LLM 不可繞過關鍵驗證閘直接產生副作用（不可直接寫關鍵資料/發致動指令，必經授權/驗證/仲裁層）。不製造第二個驅動者（同一資源單一驅動）。

## G. 溝通 / 交付
白話優先（先講人話結論再附數字，術語翻人話）。範圍詞紀律（只講最小真實範圍，個別子項過不升格成整體收官）。DEFERRED/DECISION_NEEDED/OUT_OF_SCOPE 附理由與解除條件。

## H. 機密邊界
機密/IP/商業敏感（定價、供應鏈、專有演算法、個資）不放進任何會外流的產物（公開 repo、對外文件、AI 對話公開輸出）。private repo 的白名單 gitignore 要防未來誤轉公開一次全洩。設計文件/機密不擅自 git add -f。
